# Application / Logger / UI 需求整理

## 1. 目标概述

本轮需要把 `bkui` 的应用层、日志层和 UI 基础层补齐，形成一个更稳定的最小运行骨架，方便后续继续扩展输入、动画、布局、页面管理和调试能力。

核心目标如下：

1. 增加 `Logger` 模块，提供基础日志能力。
2. 明确 `View` 与 `Box` 的职责分层。
3. 在 `Application` 中加入主循环，统一驱动可见 UI 树的 `frame` 与 `draw`。
4. 保持现有 demo 和旧接口的兼容性，避免一次性破坏已有调用方式。

## 2. Logger 模块需求

### 2.1 基础能力

- 支持日志等级：
  - `Trace`
  - `Debug`
  - `Info`
  - `Warn`
  - `Error`
  - `Fatal`
- 支持按最小等级过滤输出。
- 支持控制台彩色打印。
- 支持日志写入文件。
- 支持立即刷新，便于调试崩溃前信息。

### 2.2 设计要求

- 尽量轻量，避免复杂格式化依赖。
- 尽量内存友好，不做重型缓存。
- 线程安全，至少保证多线程写日志时不会交叉污染文本。
- 文件输出使用追加或覆盖两种模式。
- 应能由 `Application` 在初始化阶段统一接管。

### 2.3 推荐扩展

- 后续可增加：
  - 自定义 sink
  - 分类标签
  - 格式化宏
  - 平台专用控制台颜色适配
  - 异步写盘

## 3. UI 层职责整理

### 3.1 View 的定位

`ui` 目录中的 `View` 作为页面级基础单位，负责：

- 组织子视图树
- 保存 `frame`
- 控制可见性
- 驱动单帧更新
- 驱动布局
- 驱动绘制分发

建议 `View` 至少提供以下接口：

- `SetFrame / GetFrame`
- `Frame(float deltaSeconds)`
- `Draw(RenderQueue&)`
- `Measure`
- `Layout`
- `InvalidateLayout`
- `SetVisible / IsVisible`

其中：

- `Frame` 负责一帧内的逻辑推进
- `Draw` 负责输出本节点和子节点的绘制命令

### 3.2 Box 的定位

`Box` 以及其派生类，是 `View` 体系内实际承载可见内容与布局职责的基础单位。

`Box` 负责：

- 线性布局
- 子节点排布
- 可选背景绘制
- 作为其他控件的统一父类

建议后续所有常规控件都继承 `Box`，例如：

- `Label`
- `Button`
- `ImageBox`
- `ScrollBox`
- `CanvasBox`

### 3.3 当前层级建议

建议约束为：

- `View`：页面、容器树根、视图生命周期
- `Box`：可见布局基础单元
- `Box` 派生类：具体控件和组合控件

即：

`View -> Box -> 具体控件`

## 4. Application 主循环需求

### 4.1 主循环职责

`Application` 需要负责：

- 保存顶层 `View`
- 保存一帧内的 `RenderQueue`
- 维护帧编号
- 触发帧开始/帧结束事件
- 遍历所有可见顶层 `View`
- 递归调用其内部全部可见 `Box`/派生类的 `frame` 与 `draw`

### 4.2 基本流程

单帧流程建议如下：

1. 清理上一帧渲染队列
2. 触发帧开始事件
3. 遍历所有可见顶层 `View`
4. 调用 `view->Frame(deltaSeconds)`
5. 调用 `view->Draw(renderQueue)`
6. 写入本帧元数据
7. 触发帧结束事件
8. 增加帧计数

### 4.3 主循环配置

主循环至少应支持：

- 固定时间步长
- 可选真实时间步长
- 最大帧数限制
- 是否每帧清空 `RenderQueue`

## 5. 兼容性要求

为降低迁移成本，当前阶段建议保留旧式接口兼容层：

- `Render(RenderQueue&)` 内部转调 `Draw(RenderQueue&)`

这样旧 demo 仍可继续运行，新代码则逐步切换到 `Frame + Draw` 模式。

## 6. 下一步建议

在本轮完成后，推荐继续补齐以下能力：

1. `View` 父子关系与生命周期回调
2. 命中测试与事件分发
3. 焦点管理
4. 脏区域与局部重绘
5. 动画调度器
6. 统一主题与样式系统
7. 文本测量与真实字体排版接口
